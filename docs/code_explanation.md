
## `app.py`

Це головний файл вебінтерфейсу Streamlit.

У ньому створюється сторінка застосунку, поле для введення IP-адреси, домену або ASN, налаштування запуску, вкладки з результатами, JSON-вивід і фінальний AI-звіт.

`app.py` не виконує всі мережеві перевірки напряму. Він передає запит у `NetworkIntelligenceAgent`, який координує основну логіку.

## `core/config.py`

Цей файл відповідає за налаштування проєкту.

Він читає параметри з `.env`, наприклад:

- `GEMINI_API_KEY`;
- `GEMINI_MODEL`;
- таймаути;
- ключ AbuseIPDB, якщо він використовується;
- режим debug.

Це потрібно, щоб не зберігати секретні ключі напряму в коді.

## `core/schemas.py`

У цьому файлі описані структури даних, які використовуються в проєкті.

Наприклад, клас `TargetInfo` зберігає інформацію про перевірену ціль:

- початкове значення, яке ввів користувач;
- нормалізоване значення;
- тип цілі: IP, домен або ASN.

Це допомагає різним частинам програми передавати дані в одному форматі.

## `core/security.py`

Цей файл відповідає за перевірку введення користувача.

Основна функція — `normalize_target()`.

Вона приймає тільки:

- IP-адреси;
- домени;
- ASN, наприклад `AS15169`.

Якщо користувач вводить довільний текст або спробу prompt injection, наприклад `ignore previous instructions`, система не запускає аналіз, бо це не є коректною мережевою ціллю.

Також у файлі є `clip_text()`, яка обмежує довжину тексту, щоб не передавати занадто великі дані далі по системі.

## `core/agent.py`

Це головний координатор роботи системи.

Клас `NetworkIntelligenceAgent` відповідає за повний процес аналізу:

1. отримує перевірену ціль;
2. формує план інструментів;
3. запускає потрібні перевірки;
4. збирає результати;
5. отримує контекст з локальної бази знань;
6. передає дані в Gemini API;
7. повертає готовий результат у вебінтерфейс.

Важливі методи:

- `analyze_target()` — основний метод, який запускає весь аналіз;
- `_build_tool_plan()` — формує список інструментів для запуску;
- `_default_tool_plan()` — резервний план, якщо Gemini недоступний або function calling не спрацював;
- `_execute_plan()` — виконує інструменти один за одним;
- `_execute_tool()` — запускає конкретний інструмент;
- `_build_rag_query()` — формує запит для пошуку в локальній базі знань.

## `core/gemini_client.py`

Цей файл відповідає за інтеграцію з Gemini API.

Клас `GeminiNetworkClient` виконує дві основні задачі:

1. `choose_tools()` — Gemini може допомогти вибрати, які інструменти варто запустити для конкретної цілі.
2. `analyze()` — Gemini формує фінальний український аналіз на основі зібраних технічних даних.

Якщо Gemini API недоступний або ключ не вказаний, використовується fallback-логіка через `_offline_analysis()`. Це дозволяє проєкту не падати повністю, навіть якщо AI-частина тимчасово недоступна.

## `core/rag.py`

Цей файл реалізує локальний RAG-пошук.

Клас `LocalKnowledgeBase` читає Markdown-файли з папки:

`data/knowledge/`

Після цього система:

1. завантажує документи;
2. розбиває їх на фрагменти;
3. будує TF-IDF індекс;
4. шукає найбільш релевантні частини для конкретного аналізу.

Ці фрагменти додаються до контексту AI-аналізу, щоб Gemini міг краще пояснити результат.

## `tools/network_tools.py`

Цей файл містить локальні мережеві перевірки.

У ньому реалізовані такі функції:

- `ping_target()` — виконує ping-перевірку;
- `traceroute_target()` — виконує traceroute/tracert;
- `dns_lookup()` — отримує DNS-записи домену;
- `reverse_dns()` — виконує reverse DNS lookup;
- `common_port_probe()` — перевіряє популярні TCP-порти;
- `tls_certificate_info()` — читає інформацію про TLS-сертифікат.

Для ping і traceroute використовується `subprocess.run()`. Команди запускаються без `shell=True`, тому користувацький ввід не виконується як shell-код.

## `tools/external_intel.py`

Цей файл відповідає за отримання даних із зовнішніх відкритих джерел.

У ньому реалізовані такі функції:

- `resolve_first_ip()` — отримує першу IP-адресу для домену;
- `geoip_lookup()` — отримує GeoIP-інформацію;
- `rdap_lookup()` — отримує RDAP-дані;
- `ripe_network_info()` — отримує BGP/ASN-інформацію через RIPEstat;
- `dnsbl_check()` — перевіряє IP у DNSBL-базах;
- `abuseipdb_lookup()` — опційно перевіряє IP через AbuseIPDB, якщо API key налаштований.

## `data/knowledge/`

Це локальна база знань для RAG.

У ній зберігаються Markdown-файли з короткими поясненнями про:

- IP-ризики;
- DNS і TLS;
- BGP та ASN;
- prompt injection.

Ці файли можна редагувати й доповнювати без зміни основного коду.

## `docs/`

Папка `docs/` містить документацію для здачі та захисту проєкту:

- `architecture.md` — опис архітектури;
- `diagrams.md` — Mermaid-діаграми;
- `demo_script.md` — сценарій демонстрації;
- `code_explanation.md` — пояснення основних файлів коду.

## Чому це не простий чат-бот

Простий чат-бот просто передає текст користувача в AI-модель і повертає відповідь.

У цьому проєкті є повний pipeline:

1. перевірка введення;
2. визначення типу цілі;
3. запуск локальних мережевих інструментів;
4. отримання даних із зовнішніх джерел;
5. пошук у локальній базі знань;
6. AI-аналіз через Gemini API;
7. відображення результату у вебінтерфейсі.

Саме тому це потужний проєкт
